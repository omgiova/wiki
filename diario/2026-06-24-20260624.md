## Técnica: arquitetura de loop avançado do OpenClaw — 13:24

Peter Steinberger codificou soluções **reais** no OpenClaw para o "reply loop" (issue [#57263](https://github.com/openclaw/openclaw/issues/57263)), o bug onde agente fala "vou fazer" e nunca executa. Mecanismos implementados:
- **Short approval fast-path**: se o usuário diz "ok do it", o prompt é injetado com instrução pra pular planejamento e ir direto pra tool call
- **Plan-only retry com strict-agentic blocking**: se o agente só planeja por N turnos, retry automático; depois de X falhas, retorna STRICT_AGENTIC_BLOCKED — trava e reporta, não deixa loop eterno
- **Fresh context por sessão** (Ralph loop): cada sub-agente começa limpo, sem arrastar histórico de 50 mensagens
- **Verificação cruzada**: quem desenvolve não marca o próprio teste — verificador separado checa critérios de aceitação
- **QA scenarios reais**: testes automatizados que exigem tool followthrough de verdade (não aceitam "vou fazer" como resposta)

Código fonte: `src/agents/pi-embedded-runner/run.ts` e `run.incomplete-turn.ts` no repositório `openclaw/openclaw`.

## Contraste: loop Hermes vs loop OpenClaw — 13:24

O Hermes tem um loop **reativo**: usuário manda mensagem → agente processa → tools → resposta. Sem strict-agentic guard, sem fast-path de aprovação, sem retry com escalation, sem verificação cruzada. O Hermes oferece `delegate_task` (subagentes), `cronjob` (agendamento) e `execute_code` (pipelines programáticos), mas **não tem um loop engine plugável** — o loop é fixo no runtime. O Hermes docs mencionam "learning loop" que cria skills e memória, mas não tem um run loop configurável no nível do OpenClaw.
